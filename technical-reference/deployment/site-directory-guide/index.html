    Site Directory Publishing Guide¶

    Overview¶

    The site/ directory contains the built static HTML files generated by MkDocs. This guide explains how to handle this directory for different deployment scenarios.

    Key Principle: Don't Commit site/¶

    The site/ directory should NEVER be committed to your main branch.

    Why:

    • It's generated output (like compiled binaries)
    • Creates massive, meaningless diffs
    • Bloats repository size
    • Can cause merge conflicts
    • GitHub Pages can build it automatically

    Deployment Options¶

    Option 1: GitHub Pages with GitHub Actions (Recommended)¶

    Let MkDocs build and deploy automatically:

    # .github/workflows/deploy-docs.yml
name: Deploy Documentation

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.x'

      - name: Install dependencies
        run: |
          pip install -r requirements.txt

      - name: Build site
        run: mkdocs build

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v2
        with:
          path: ./site

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v2

    Option 2: MkDocs gh-deploy Command¶

    Use MkDocs built-in GitHub Pages deployment:

    # Builds and pushes to gh-pages branch
mkdocs gh-deploy

# With custom commit message
mkdocs gh-deploy -m "Update documentation [skip ci]"

# Clean build (removes old files)
mkdocs gh-deploy --clean

# Specific remote/branch
mkdocs gh-deploy --remote-name origin --remote-branch gh-pages

    How it works:

    1. Builds site to site/ directory
    2. Creates/updates gh-pages branch
    3. Commits only the built files
    4. Force pushes to gh-pages
    5. GitHub Pages serves from this branch

    Option 3: Manual Deployment to Other Hosts¶

    For non-GitHub hosting:

    # Build the site
mkdocs build --clean

# Deploy to server via rsync
rsync -avz --delete site/ user@server:/var/www/druids-docs/

# Deploy to S3
aws s3 sync site/ s3://your-bucket/ --delete

# Deploy to Netlify
netlify deploy --dir=site --prod

# Deploy to Vercel
vercel --prod site/

    Development Workflow¶

    Local Development¶

    # Start dev server (doesn't create site/)
mkdocs serve --dev-addr 127.0.0.1:8001

# Live reload on changes
mkdocs serve --livereload

# Strict mode (fails on warnings)
mkdocs serve --strict

    Preview Production Build¶

    # Build site locally
mkdocs build

# Serve the built site
python -m http.server -d site 8002

# Or use any static server
npx serve site -p 8002

    Clean Builds¶

    # Remove old site directory
rm -rf site/

# Build with clean flag
mkdocs build --clean

# Or use Makefile
make clean build

    Configuration¶

    Set Site Directory (if needed)¶

    # mkdocs.yml
site_dir: build  # Change output directory name

    GitHub Pages Configuration¶

    # mkdocs.yml
# For project pages (user.github.io/project)
site_url: https://username.github.io/druids-wiki/

# For organization pages (org.github.io)
site_url: https://ksbc.github.io/

    Custom Domain¶

    Create docs/CNAME file:

    druids.example.com

    Troubleshooting¶

    Site Directory in Git¶

    If accidentally committed:

    # Remove from tracking
git rm -r --cached site/

# Add to .gitignore
echo "site/" >> .gitignore

# Commit changes
git add .gitignore
git commit -m "chore: remove site directory from tracking"

    GitHub Pages Not Updating¶

    1. Check GitHub Pages settings
    2. Verify branch name (gh-pages vs main)
    3. Check for build errors in Actions
    4. Clear browser cache
    5. Wait 10 minutes (CDN propagation)

    Build Errors¶

    # Check for errors
mkdocs build --strict

# Verbose output
mkdocs build --verbose

# Check configuration
mkdocs build --config-file mkdocs.yml

    Large Site Directory¶

    # Check size
du -sh site/

# Find large files
find site -type f -size +1M -exec ls -lh {} \;

# Optimize images before building
find docs -name "*.png" -exec optipng {} \;
find docs -name "*.jpg" -exec jpegoptim {} \;

    Best Practices¶

    1. Use CI/CD for Deployment¶

    Never deploy from local machine:

    • Ensures consistent environment
    • Automated quality checks
    • No accidental local files
    • Clear deployment history

    2. Protect Main Branch¶

    # Branch protection rules:
- Require PR reviews
- Run build checks before merge
- No direct pushes to main
- No force pushes

    3. Preview Deployments¶

    For pull requests:

    # Deploy preview for PRs
- name: Deploy Preview
  if: github.event_name == 'pull_request'
  run: |
    mkdocs build
    netlify deploy --dir=site --alias=pr-${{ github.event.number }}

    4. Version Documentation¶

    For multiple versions:

    # Use mike for versioning
pip install mike

# Deploy version
mike deploy --push --update-aliases 1.0 latest

# List versions
mike list

    Security Considerations¶

    Never Commit Sensitive Data¶

    Before deploying, check:

    # Scan for secrets
grep -r "password\|api_key\|token" site/

# Check for personal info
grep -r "@gmail\|@yahoo" site/

# Run privacy audit
./scripts/privacy-audit.sh

    Use Environment Variables¶

    # mkdocs.yml
extra:
  analytics:
    provider: google
    property: !ENV GOOGLE_ANALYTICS_ID

    Secure Deployment¶

    # Use deployment tokens, not personal tokens
export GITHUB_TOKEN="${{ secrets.DEPLOY_TOKEN }}"

# Restrict permissions
chmod 600 ~/.ssh/deploy_key

    Quick Reference¶

    # Local development
mkdocs serve

# Build for production
mkdocs build --clean

# Deploy to GitHub Pages
mkdocs gh-deploy

# Check for issues
mkdocs build --strict

# Clean up
rm -rf site/

    Makefile Example¶

    # Makefile
.PHONY: serve build deploy clean

serve:
 mkdocs serve --dev-addr 127.0.0.1:8001

build: clean
 mkdocs build --strict

deploy: build
 mkdocs gh-deploy --clean

clean:
 rm -rf site/

check:
 mkdocs build --strict
 ./scripts/privacy-audit.sh

    Remember: The site/ directory is OUTPUT, not SOURCE. Keep it out of version control!

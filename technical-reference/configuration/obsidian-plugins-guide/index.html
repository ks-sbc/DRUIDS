    DRUIDS Obsidian Plugins Reference Guide¶

    Overview¶

    This guide documents all Obsidian plugin configurations for the DRUIDS infrastructure. Each plugin has been configured to support democratic centralist workflows, enforce security policies, and automate organizational processes.

    Plugin Categories¶

    üîí Security & Compliance¶

    • Obsidian Git - Automated backups with security awareness
    • Linter - Document standard enforcement
    • GPGCrypt - L2 vault encryption
    • Style Settings - Visual security indicators

    üìã Task & Project Management¶

    • Tasks - Democratic workflow statuses
    • Kanban - Visual task boards
    • Projects - Project tracking with Dataview
    • MetaEdit - Frontmatter management

    ü§ñ Automation & Workflows¶

    • QuickAdd - Workflow macros
    • Templater - Advanced templates
    • Shell Commands - System integration
    • Dataview - Dynamic queries

    üìù Content Creation¶

    • Advanced Slides - Presentations
    • Table Editor - Table management
    • Notion-like Tables - Enhanced tables
    • Tag Wrangler - Tag management
    • TagFolder - Tag navigation

    üé® Appearance & UI¶

    • Style Settings - Theme customization
    • CSS Editor - Custom styles
    • Custom CSS - Additional styling

    Quick Reference¶

    Plugin Purpose Key Features Security Level

    Configuration Standards¶

    Security Principles¶

    1. Least Privilege: Disable features unless explicitly needed
    2. Defense in Depth: Multiple security layers
    3. Visual Awareness: Color-coded security indicators
    4. Automated Validation: Pre-commit checks

    Workflow Integration¶

    1. Democratic Centralism: 6-stage workflow support
    2. Task Lifecycle: Proposal ‚Üí Implementation ‚Üí Summation
    3. Voting Mechanisms: Track decisions and consensus
    4. Audit Trail: Document all changes

    Performance Optimization¶

    1. Selective Features: Only enable necessary functions
    2. Efficient Queries: Limit Dataview refresh rates
    3. Batch Operations: Group similar tasks
    4. Cache Management: Regular cleanup schedules

    File Locations¶

    All plugin configurations are stored in:

    /repos/shared/obsidian/plugins/{plugin-name}/data.json

    These are symlinked to individual vaults:

    /vault/public/.obsidian ‚Üí /repos/shared/obsidian
/vault/member/.obsidian ‚Üí /repos/shared/obsidian

    Maintenance¶

    Daily Checks¶

    • Git auto-backup functioning
    • No security warnings in logs
    • Visual indicators displaying correctly

    Weekly Tasks¶

    • Review Linter validation reports
    • Check for plugin updates
    • Audit shell command executions

    Monthly Reviews¶

    • Security classification accuracy
    • Workflow efficiency metrics
    • Plugin performance impact

    Troubleshooting¶

    Common Issues¶

    1. Plugin Not Loading
    2. Check manifest.json exists
    3. Verify main.js is present

    4. Reload Obsidian

    5. Configuration Not Applied

    6. Verify symlinks are intact
    7. Check file permissions

    8. Clear Obsidian cache

    9. Security Indicators Missing

    10. Enable Style Settings
    11. Check CSS snippet is active
    12. Verify theme compatibility

    Debug Commands¶

    # Check plugin status
ls -la /repos/shared/obsidian/plugins/

# Verify symlinks
readlink /vault/*/obsidian

# Test configurations
node /repos/shared/scripts/js/validate-plugins.js

    Security Considerations¶

    ‚ö†Ô∏è Critical Security Rules¶

    1. Never store credentials in plugin configurations
    2. Disable JavaScript execution in Dataview
    3. Restrict shell command access
    4. Audit all external plugin installations
    5. Regular security scans of configurations

    L2 Vault Restrictions¶

    The L2 (cadre) vault has additional restrictions:

    • GPGCrypt mandatory for all files
    • No cloud sync plugins allowed
    • Restricted shell command set
    • Audit logging enabled

    Best Practices¶

    Configuration Management¶

    • Always test in one vault before propagating
    • Document all customizations
    • Use version control for configurations
    • Regular backup of settings

    Performance¶

    • Monitor plugin impact on startup time
    • Disable unused features
    • Optimize Dataview queries
    • Limit real-time processing

    Security¶

    • Regular credential rotation
    • Audit plugin permissions
    • Monitor for suspicious activity
    • Enforce least privilege

    Updates and Maintenance¶

    Plugin Updates¶

    1. Test updates in isolated environment
    2. Review changelog for breaking changes
    3. Backup configurations before updating
    4. Verify security settings post-update

    Configuration Changes¶

    1. Document reason for change
    2. Test in development vault
    3. Apply to all vaults via shared config
    4. Verify functionality across vaults

    Support¶

    For issues or questions:

    1. Check plugin-specific documentation
    2. Review troubleshooting section
    3. Consult DRUIDS development team
    4. Submit issue to repository

    Dataview Security Guidelines¶

    Overview¶

    Dataview is a powerful Obsidian plugin that allows querying and displaying data from your vault. However, its JavaScript execution capabilities present potential security risks. This guide outlines best practices for using Dataview securely within the DRUIDS infrastructure.

    Security Considerations¶

    JavaScript Execution Risks¶

    When JavaScript execution is enabled in Dataview:

    • Arbitrary code can be executed within queries
    • Access to the Obsidian API and file system
    • Potential for malicious scripts in shared queries
    • Risk of data exfiltration or vault corruption

    Current Configuration¶

    For DRUIDS, we've made the following security decisions:

    1. JavaScript Queries: Disabled by default
    2. Inline JavaScript: Disabled by default
    3. HTML Rendering: Disabled to prevent XSS attacks
    4. Safe Mode: Only DQL (Dataview Query Language) allowed

    Safe Query Patterns¶

    Use DQL Instead of JavaScript¶

    ❌ Avoid JavaScript queries:

    ```dataviewjs
// This allows arbitrary code execution
const pages = dv.pages("#meeting");
for (let page of pages) {
    // Potential security risk
    await app.vault.modify(page.file, "modified content");
}

    ✅ **Use DQL queries instead:**
```markdown
```dataview
TABLE 
    file.ctime as Created,
    security_level as "Security"
FROM #meeting
WHERE date(file.ctime) >= date(now) - dur(7 days)
SORT file.ctime DESC

    ### Whitelisted Query Patterns

These patterns are considered safe for DRUIDS usage:

#### 1. Organization Metrics Dashboard
```markdown
```dataview
TABLE WITHOUT ID
    length(rows) as Count,
    security_level as "Security Level"
FROM ""
WHERE file.name != this.file.name
GROUP BY security_level

    #### 2. Task Tracking
```markdown
```dataview
TASK
WHERE !completed AND contains(tags, "#todo")
GROUP BY file.link

    #### 3. Meeting Overview
```markdown
```dataview
TABLE 
    meeting_date as "Date",
    attendees as "Attendees",
    length(action_items) as "Actions"
FROM #meeting
WHERE meeting_date >= date(now) - dur(30 days)
SORT meeting_date DESC

    #### 4. Member Activity
```markdown
```dataview
LIST
FROM #member-contribution
WHERE author = "Current Member"
SORT file.mtime DESC
LIMIT 10

    #### 5. Project Status
```markdown
```dataview
TABLE 
    status as Status,
    due_date as "Due Date",
    assigned_to as "Assigned"
FROM #project
WHERE status != "completed"
SORT due_date ASC

    ## Restricted Operations

The following operations should NEVER be used in shared queries:

### 1. File System Modifications
```javascript
// FORBIDDEN
await app.vault.create(...)
await app.vault.modify(...)
await app.vault.delete(...)
await app.vault.rename(...)

    2. External Network Calls¶

    // FORBIDDEN
fetch('https://external-api.com/...')
XMLHttpRequest(...)

    3. Plugin Manipulation¶

    // FORBIDDEN
app.plugins.disablePlugin(...)
app.plugins.enablePlugin(...)

    4. Credential Access¶

    // FORBIDDEN
localStorage.getItem(...)
sessionStorage.getItem(...)

    Security Best Practices¶

    For Query Authors¶

    1. Use DQL Exclusively: Avoid JavaScript queries unless absolutely necessary
    2. Validate Input: When using dynamic queries, validate all inputs
    3. Limit Scope: Query only the data you need
    4. Document Queries: Comment complex queries for review
    5. Test in Isolation: Test queries in a separate vault first

    For Vault Administrators¶

    1. Disable JavaScript: Keep JavaScript execution disabled unless required
    2. Review Shared Queries: Audit all queries before implementation
    3. Monitor Performance: Watch for queries that consume excessive resources
    4. Regular Audits: Periodically review all Dataview queries in the vault
    5. Education: Train members on secure query practices

    Configuration Recommendations¶

    High Security (Recommended)¶

    {
  "enableJavaScriptQueries": false,
  "enableInlineJavaScriptQueries": false,
  "enableDataviewJs": false,
  "enableInlineDataviewJs": false,
  "allowHtml": false
}

    Medium Security¶

    {
  "enableJavaScriptQueries": false,
  "enableInlineJavaScriptQueries": false,
  "enableDataviewJs": true,
  "enableInlineDataviewJs": false,
  "allowHtml": false
}

    Development Only¶

    {
  "enableJavaScriptQueries": true,
  "enableInlineJavaScriptQueries": false,
  "enableDataviewJs": true,
  "enableInlineDataviewJs": false,
  "allowHtml": false
}

    Monitoring and Auditing¶

    Query Audit Checklist¶

    • Query uses only DQL (no JavaScript)
    • No file system operations
    • No external network calls
    • No credential access attempts
    • Query scope is limited
    • Performance impact assessed
    • Security classification appropriate
    • Documentation provided

    Regular Security Reviews¶

    1. Weekly: Review new queries added to vault
    2. Monthly: Audit all existing queries
    3. Quarterly: Update security guidelines
    4. Annually: Full security assessment

    Incident Response¶

    If a malicious or problematic query is discovered:

    1. Immediate Actions
    2. Disable the query
    3. Document the issue

    4. Assess impact

    5. Investigation

    6. Identify query source
    7. Check for data exposure

    8. Review audit logs

    9. Remediation

    10. Remove malicious code
    11. Update security settings

    12. Notify affected members

    13. Prevention

    14. Update guidelines
    15. Implement additional controls
    16. Provide additional training

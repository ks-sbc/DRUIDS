    Tag Hierarchy System¶

    Overview¶

    The DRUIDS tag hierarchy provides a structured categorization system for organizing knowledge across security tiers. This reference documents all available tags and their proper usage.

    Core Tag Categories¶

    Organizational Tags¶

    • #DRUIDS - Main system tag
    • #KSBC - Kansas Socialist Book Club specific
    • #Meeting - Meeting-related documents
    • #Proposal - Proposal documents
    • #Task - Task management

    Security Classification Tags¶

    • #Security - Security-related content
    • #L0-Public - Public tier content
    • #L1-Member - Member tier content
    • #L2-Cadre - Cadre tier content (FORBIDDEN to AI)

    Document Type Tags¶

    • #Tutorial - Learning-oriented guides
    • #HowTo - Task-oriented guides
    • #Reference - Technical specifications
    • #Explanation - Conceptual understanding

    Workflow Tags¶

    • #Workflow - Workflow documentation
    • #GitFlow - Git workflow specific
    • #Democratic-Centralism - Democratic centralist processes
    • #Automation - Automated processes

    Technical Tags¶

    • #Obsidian - Obsidian-specific content
    • #MkDocs - Documentation site related
    • #Configuration - Configuration guides
    • #Development - Development resources

    Tag Usage Guidelines¶

    1. Primary Tags¶

    Every document should have at least one primary tag:

    • Document type tag (Tutorial, HowTo, Reference, Explanation)
    • Security classification tag (L0, L1, L2)

    2. Secondary Tags¶

    Add relevant secondary tags for:

    • Subject matter (Meeting, Proposal, Security)
    • Technical area (Obsidian, Git, Configuration)
    • Organizational context (DRUIDS, KSBC)

    3. Tag Combinations¶

    Common tag combinations:

    • Meeting minutes: #Meeting #L1-Member #Reference
    • Security guides: #Security #HowTo #DRUIDS
    • Technical docs: #Reference #Configuration #Obsidian

    Tag Validation¶

    Tags are validated through:

    1. MkDocs tags plugin configuration
    2. Obsidian tag pane organization
    3. Git pre-commit hooks

    Best Practices¶

    1. Consistency: Use standardized tag names
    2. Hierarchy: Apply both broad and specific tags
    3. Security: Always include security classification
    4. Searchability: Tag for how users will search

    Forbidden Combinations¶

    Never combine:

    • L0 content with L2 tags
    • Public tutorials with member-only content
    • Multiple security tier tags on one document

    Tag Maintenance¶

    Regular maintenance tasks:

    1. Review tag usage monthly
    2. Consolidate redundant tags
    3. Update tag documentation
    4. Train new members on tagging

    Integration Points¶

    Obsidian¶

    • Tags appear in tag pane
    • Used for graph view filtering
    • Enable quick filtering

    MkDocs¶

    • Tags create index pages
    • Enable cross-referencing
    • Support tag clouds

    Git¶

    • Tags in frontmatter
    • Validated on commit
    • Used for automation

    Removing Secrets from Git History¶

    Overview¶

    This guide documents the procedure for completely removing exposed credentials from Git history after they have been accidentally committed. This is a critical security procedure that must be performed immediately upon discovery of exposed secrets.

    Prerequisites¶

    • Git installed and configured
    • Write access to the repository
    • Backup of the repository (essential before history rewriting)
    • All team members notified of the upcoming force push

    Procedure¶

    1. Immediate Mitigation¶

    First, remove the secret from the current version:

    # Edit the file to remove the secret
git add <file>
git commit -m "[SECURITY] Remove exposed credential"

    2. Identify Affected Commits¶

    Find all commits that contain the exposed secret:

    # Search for the secret in history
git log -S"<secret_string>" --all --oneline

# Alternative: use git grep on all branches
git grep "<secret_string>" $(git rev-list --all)

    3. Clean Git History¶

    Option A: Using BFG Repo-Cleaner (Recommended)¶

    BFG is faster and simpler than git filter-branch:

    # Download BFG
wget https://repo1.maven.org/maven2/com/madgag/bfg/1.14.0/bfg-1.14.0.jar

# Create a fresh clone
git clone --mirror <repo_url> repo-mirror
cd repo-mirror

# Create secrets.txt with patterns to remove
echo "sk-proj-*" > secrets.txt
echo "*api[_-]?key*=*" >> secrets.txt

# Run BFG to remove secrets
java -jar ../bfg-1.14.0.jar --replace-text secrets.txt

# Clean up
git reflog expire --expire=now --all && git gc --prune=now --aggressive

# Push cleaned history
git push --force

    Option B: Using git filter-repo (Alternative)¶

    # Install git-filter-repo
pip install git-filter-repo

# Create expressions file
echo "regex:sk-proj-[A-Za-z0-9_-]+==>" > expressions.txt

# Run filter-repo
git filter-repo --replace-text expressions.txt

# Force push
git push --force --all
git push --force --tags

    Option C: Using native git filter-branch (Last Resort)¶

    # Remove file containing secret from all history
git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch path/to/file' \
  --prune-empty --tag-name-filter cat -- --all

# Or replace secret in file
git filter-branch --tree-filter \
  'find . -type f -exec sed -i "s/EXPOSED_SECRET/REDACTED/g" {} +' \
  -- --all

    4. Clean Local Repository¶

    # Remove original refs
rm -rf .git/refs/original/

# Expire reflog
git reflog expire --expire=now --all

# Garbage collect
git gc --prune=now --aggressive

    5. Force Push Changes¶

    WARNING: This will rewrite history for all users!

    # Push all branches
git push --force --all

# Push all tags
git push --force --tags

    6. Notify Team Members¶

    All team members must:

    # Backup local changes
git stash

# Delete and re-clone
cd ..
rm -rf <repo_name>
git clone <repo_url>
cd <repo_name>

# Or force update existing repo (not recommended)
git fetch --all
git reset --hard origin/<branch>

    Security Best Practices for Credentials¶

    Never Commit Credentials¶

    1. Use Environment Variables
    // Instead of:
const apiKey = "sk-proj-secret123";

// Use:
const apiKey = process.env.OPENAI_API_KEY;
    1. Use .gitignore
    # Add to .gitignore
.env
.env.local
*.key
*secrets*
config/credentials.json
    1. Use Git Hooks Create .git/hooks/pre-commit:
    #!/bin/bash
# Prevent commits with secrets
if git diff --cached | grep -E "(api[_-]?key|secret|password|token)" | grep -v "REDACTED"; then
  echo "Error: Possible secret detected in commit"
  exit 1
fi

    For Obsidian Plugin Configurations¶

    1. QuickAdd AI Configuration
    2. Leave apiKey field empty: "apiKey": ""
    3. Users should set via plugin settings UI

    4. Never commit with populated API keys

    5. Environment Variable Loading

    // In plugin settings
const apiKey = process.env.QUICKADD_OPENAI_KEY || "";
    1. Secure Storage
    2. Use Obsidian's built-in vault for sensitive settings
    3. Exclude from git: .obsidian/plugins/*/secrets.json

    Verification¶

    After cleaning history:

    # Verify secret is gone
git log --all --grep="<secret_pattern>"
git grep "<secret_pattern>" $(git rev-list --all)

# Check file history
git log -p path/to/file | grep -i "secret"

# Verify on remote
git clone <repo_url> test-clean
cd test-clean
git log --all --oneline | xargs git grep "<secret_pattern>"

    Additional Security Measures¶

    1. Rotate Compromised Credentials
    2. Immediately revoke/regenerate any exposed keys

    3. Update all systems using the old credentials

    4. Audit Access Logs

    5. Check if exposed credentials were used

    6. Review API access logs for suspicious activity

    7. Enable Security Scanning

    # .github/workflows/security.yml
name: Security Scan
on: [push, pull_request]
jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run git-secrets
        run: |
          git secrets --install
          git secrets --register-aws
          git secrets --scan
